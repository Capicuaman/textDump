Claude Code Commands

Official Built-in Slash Commands & Custom Command Guide

What Are Slash Commands?

Slash commands are special instructions that start with / in Claude Code (the CLI tool). They provide quick access to functionality like clearing conversations, managing tasks, configuring settings, and more.

Pro Tip: Type /help at any time to see all available commands in your CLI session.
Important: This guide documents ONLY the actual built-in commands that exist in Claude Code. Many commands you might expect (like /commit or /diff) are NOT built-in, but you can create them as custom commands (see the Custom Commands section below).

Built-in Session Commands

/help

Get Help and View Available Commands

Displays all available slash commands with brief descriptions.

Use Case: You're new to Claude Code and want to discover what commands are available.
Example:
/help - Shows complete command list
/clear

Clear Conversation History

Starts a fresh conversation, clearing all previous context. Useful when switching to a completely different task.

Use Case: You've finished debugging one feature and want to start fresh on a completely unrelated task without context pollution.
Warning: This removes all conversation history. Make sure you've saved any important information first.
/quit or /exit

Exit Claude Code

Cleanly exits the CLI session.

Use Case: You're done with your coding session and want to close Claude Code properly.
/status

Show System Status

Displays version, model, account information, and connectivity status.

Use Case: Check which model you're currently using or verify your API connection.
Note: This shows system/account status, NOT git status. For git status, use natural language ("show git status") or create a custom command.

Built-in Task Management Commands

/tasks

View Active Background Tasks

Lists all running background tasks including agents, shell processes, and async operations.

Use Case: You launched multiple agents in parallel and want to check their status or get their task IDs.
Example Workflow:
1. Start background task: "Run tests in background"
2. Check status: /tasks
3. View output: Use task ID from /tasks to retrieve results
/model

Switch AI Model

Change between different Claude models (Opus, Sonnet, Haiku) for the current session.

Use Case: Switch to Haiku for simple file searches to save costs, then switch back to Sonnet for complex refactoring.
Models:
/model sonnet - Balanced (recommended)
/model opus - Most capable
/model haiku - Fastest and cheapest

Built-in Code Review Commands

/review

Request Code Review

Triggers Claude to review your code changes and provide feedback.

Use Case: You've made changes and want a comprehensive code review before committing.
Note: This may be a custom command in your setup. Check .claude/commands/ to see if it's customized for your workflow.
/security-review

Security Review of Changes

Performs a security-focused review of pending changes on the current branch.

Use Case: Before merging security-sensitive code, check for common vulnerabilities.
Checks For:
- SQL injection
- XSS vulnerabilities
- Authentication/authorization issues
- Exposed secrets
/pr-comments

View Pull Request Comments

Displays comments from a pull request for review and discussion.

Use Case: Review feedback on your PR without leaving the CLI.

Built-in Configuration Commands

/settings

Open Settings

Opens settings interface to configure Claude Code behavior, API keys, model preferences, etc.

Use Case #1: Set default model to Haiku for cost savings on simple tasks.
Use Case #2: Configure hooks that run before/after certain actions.
Use Case #3: Set up MCP servers for extended functionality.

Natural Language Alternative

You Don't Always Need Slash Commands!
Claude understands natural language requests. Instead of looking for a /commit command, just say:
  • "Create a git commit with my changes"
  • "Show me git status and diff"
  • "Create a pull request for this branch"
  • "Run the test suite"
  • "Add these files to context: src/app.js src/utils.js"
Claude will execute the appropriate bash commands or tool calls automatically.

Creating Custom Slash Commands

Since many useful commands (like /commit, /diff, /pr) don't exist as built-ins, you can create your own custom commands. Custom commands are stored in .claude/commands/ directory as markdown files.

Command File Structure

Create a file .claude/commands/[command-name].md with this structure:

--- allowed-tools: Bash(git:*), Read, Edit argument-hint: [optional arguments] description: Brief description of what this command does --- ## Context Provide any context needed (can use !`command` to run bash commands): - Current status: !`git status` - Current diff: !`git diff` ## Your task Describe what Claude should do when this command is invoked. Use $ARGUMENTS to reference any arguments passed to the command.

Frontmatter Fields Explained

Field Description Example
allowed-tools Tools Claude can use for this command (optional, restricts access) Bash(git:*), Read, Grep
argument-hint Shown to user as hint for arguments [message], [branch-name]
description Short description shown in /help Create a git commit

Using Dynamic Context with !`command`

You can run bash commands in the Context section using !`command` syntax. The output is included when the command runs:

## Context - Current git status: !`git status` - Staged changes: !`git diff --cached` - Last 5 commits: !`git log --oneline -5` - Current branch: !`git branch --show-current`

Accessing Arguments with $ARGUMENTS

When users invoke /command arg1 arg2, use $ARGUMENTS to access those arguments:

## Your task Create a git commit with message: $ARGUMENTS If no message provided in $ARGUMENTS, analyze the changes and create an appropriate commit message.

Advanced Custom Command Patterns

Chaining Multiple Commands

You can create commands that execute multiple operations in sequence:

--- description: Run tests, build, and deploy --- ## Your task Execute the following steps in order: 1. Run test suite - if tests fail, STOP and report 2. Build production assets - if build fails, STOP and report 3. Create git commit with changes 4. Push to remote 5. Deploy to staging environment 6. Report final status and URLs

Conditional Logic in Commands

Use conditional instructions for smart behavior:

## Your task 1. Check if $ARGUMENTS is provided 2. If YES: use $ARGUMENTS as commit message 3. If NO: analyze git diff and generate message automatically 4. If staged changes exist: commit staged only 5. If no staged changes: add all modified files then commit

Project-Specific Commands

Create commands tailored to your project's workflow:

--- description: BILAN marketing content workflow --- ## Context - Project: BILAN electrolyte powder business - Current marketing campaigns: !`ls 01_PROJECTS/BILAN/MARKETING/campaigns/` ## Your task 1. Create new marketing content in $ARGUMENTS campaign 2. Update FAQ database if content is FAQ-worthy 3. Generate VEO video prompts if applicable 4. Update social media calendar 5. Commit changes with descriptive message

Command Best Practices

1. Start with Context
Use !`commands` to gather current state before acting. This gives Claude the information needed to make smart decisions.
2. Be Explicit About Behavior
Don't assume Claude knows what to do. Clearly describe each step in the "Your task" section.
3. Handle Edge Cases
Include instructions for what to do if things go wrong (no files found, tests fail, etc.).
4. Restrict Tools When Needed
Use allowed-tools to limit what Claude can do in sensitive commands. For example, a read-only command shouldn't have Write or Bash access.
5. Make Commands Composable
Create small, focused commands that do one thing well. You can combine them in workflows rather than creating monolithic commands.
6. Test Your Commands
After creating a command, test it with different arguments and edge cases to ensure it behaves as expected.

Troubleshooting Custom Commands

Command Not Found
Ensure your .md file is in .claude/commands/ and has proper frontmatter with at least a description field.
Permission Errors
If Claude says it can't execute bash commands, check that allowed-tools includes the necessary tools. Consider adding specific command patterns like Bash(git:*).
Context Commands Not Running
The !`command` syntax only works in the Context section. It won't execute in "Your task" - use explicit instructions there instead.
Arguments Not Working
Make sure you're using $ARGUMENTS (uppercase) not $arguments. The variable is case-sensitive.
